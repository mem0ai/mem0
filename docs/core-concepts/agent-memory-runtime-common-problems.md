Документ фиксирует наиболее частые и наиболее дорогие по последствиям проблемы памяти в агентных системах и показывает, как `Agent Memory Runtime` их решает, частично решает или пока сознательно не берет в `v1`.

## Статус

- Документ описывает именно problem-space, а не roadmap целиком.
- Ниже статус решений дается честно относительно текущего состояния проекта.
- Формулировки `решаем`, `частично`, `не решаем пока` означают текущее состояние, а не маркетинговое обещание.

## Как читать статусы

- `решаем` — в проекте уже есть рабочий механизм, закрывающий проблему в первом практическом объеме
- `частично решаем` — есть базовый контур, но качество или полнота решения пока ограничены
- `не решаем пока` — проблема признана, но не входит в текущий рабочий контур

## 1. Агент забывает все между сессиями

### Проблема

После завершения одной сессии агент стартует как будто “с нуля”, даже если уже видел проект, архитектуру, предпочтения пользователя или прошлые решения.

### Почему это больно

- агент снова задает те же вопросы
- теряет continuity работы
- не накапливает реальную компетентность по проекту
- стоимость работы растет с каждой новой сессией

### Как решаем

- принимаем события через `POST /v1/events`
- формируем `episodes`
- консолидируем полезное в `memory_units`
- используем `recall` перед следующей работой агента

### Статус

- `решаем`

## 2. Краткосрочный шум засоряет долгосрочную память

### Проблема

Агент сохраняет все подряд: промежуточные мысли, мелкие статусы, временные команды, случайный мусор из диалога.

### Почему это больно

- память быстро превращается в свалку
- релевантность recall падает
- в контекст попадает лишнее
- полезные факты теряются на фоне шума

### Как решаем

- разделяем `session-space`, `project-space`, `agent-core`, `shared-space`
- сначала сохраняем событие и эпизод, а не сразу “финальную память”
- перевод в long-term идет через отдельный consolidation pipeline
- lifecycle baseline делает `decay`, `archive`, `evict`

### Статус

- `решаем`

### Ограничение

- шумофильтрация пока не является семантически глубокой и не использует полноценную защиту от memory poisoning

## 3. Агент помнит факты, но не умеет отделять short-term от long-term

### Проблема

Во многих системах “память” — это один общий мешок, где временные детали и устойчивые знания лежат без явной границы.

### Почему это больно

- временное загрязняет постоянное
- постоянное не получает приоритет при retrieval
- забывание становится опасным и хаотичным

### Как решаем

- ввели явные `memory spaces`
- `session-space` работает как hot-layer
- consolidation определяет `kind` и `scope`
- lifecycle применяет разные правила к `session-space` и long-term spaces

### Статус

- `решаем`

## 4. Память есть, но retrieval возвращает не то

### Проблема

Классическая беда memory systems: данные хранятся, но при реальном запросе агент получает либо мусор, либо что-то слишком общее, либо не получает нужное вообще.

### Почему это больно

- memory technically exists, but product value is near zero
- агент начинает игнорировать память
- контекст становится длиннее, а не полезнее

### Как решаем

- используем `POST /v1/recall`, а не “просто search”
- делаем `scope resolution`
- собираем кандидатов из активных spaces
- ранжируем по lexical overlap, importance, session affinity, recency
- добавили `usefulness feedback loop`
- возвращаем не raw results, а `MemoryBrief`

### Статус

- `частично решаем`

### Что еще не закрыто

- ranking пока еще не является сильным semantic/hybrid retrieval движком
- полноценная mem0-based retrieval orchestration пока только как integration seam, а не как основной production path

## 5. Агент не понимает, какая память реально помогла

### Проблема

Большинство memory systems сохраняют и извлекают записи, но почти не учатся на том, что было полезно, а что нет.

### Почему это больно

- retrieval не улучшается со временем
- память не адаптируется под реальную работу агента
- сложно стабильно поднимать качество

### Как решаем

- recall trace возвращает `selected_episode_ids`
- есть `POST /v1/recall/feedback`
- feedback пишется в `audit_log`
- ranking учитывает positive/negative usefulness signals

### Статус

- `решаем`

### Ограничение

- это пока rule-based improvement loop, а не полноценная learned ranking system

## 6. Память дублируется и распухает

### Проблема

Один и тот же факт сохраняется много раз в слегка разных формулировках.

### Почему это больно

- long-term memory разрастается без роста ценности
- retrieval начинает возвращать дубликаты
- сжимаемость контекста ухудшается

### Как решаем

- есть `merge_key`
- consolidation умеет `create / merge`
- повторные эпизоды с тем же ядром смысла схлопываются в один `memory_unit`

### Статус

- `частично решаем`

### Что еще не закрыто

- пока это ближе к normalized deduplication, чем к настоящему semantic dedup/contradiction resolution

## 7. Агент продолжает опираться на устаревшую память

### Проблема

Старые решения, устаревший стек, прошлые настройки и уже неактуальные факты продолжают подмешиваться в контекст.

### Почему это больно

- агент принимает решения на старой реальности
- снижается доверие к памяти
- появляются ложные рекомендации и regression in behavior

### Как решаем

- lifecycle manager считает freshness
- есть `decay`
- есть `archive`
- есть `evict`
- retrieval учитывает recency

### Статус

- `частично решаем`

### Что еще не закрыто

- нет полноценного механизма semantic invalidation
- нет сильной модели противоречий вида “старый факт заменен новым”

## 8. Система не умеет забывать

### Проблема

Очень частый анти-паттерн: “память” только накапливается и никогда осмысленно не стареет.

### Почему это больно

- стоимость и шум постоянно растут
- retrieval деградирует
- long-term store со временем становится токсичным для качества

### Как решаем

- есть lifecycle pipeline
- есть `TTL`-подобное поведение для session-like слоев
- есть `decay`
- есть `archive`
- есть `evict`

### Статус

- `решаем`

### Ограничение

- forgetting пока еще policy baseline, не полноценная когнитивная модель забывания

## 9. Общая память нескольких агентов протекает или, наоборот, не работает

### Проблема

В multi-agent системах типовой провал выглядит так:

- либо агенты не могут нормально делиться знаниями
- либо начинают видеть то, что должно оставаться приватным

### Почему это больно

- coordination ломается
- появляется memory leakage
- shared work не масштабируется

### Как решаем

- есть `isolated` и `shared` namespace modes
- есть `shared-space`
- межагентная видимость разрешена только там, где это допустимо
- `agent-core` остается приватным
- adapter layer валидирует source-system ownership

### Статус

- `решаем`

### Ограничение

- живая продовая интеграция пока полностью завершена не для всех клиентов одновременно

## 10. У памяти нет объяснимости

### Проблема

Агенту что-то подмешали в контекст, но непонятно:

- откуда это взялось
- почему это выбрали
- что было проигнорировано

### Почему это больно

- почти невозможно улучшать retrieval
- трудно дебажить странное поведение агента
- сложно строить доверие к memory runtime

### Как решаем

- есть `audit_log`
- есть `RecallTrace`
- observability layer экспортирует counters и queue/job state
- selected episode refs теперь возвращаются наружу

### Статус

- `частично решаем`

### Что еще не закрыто

- пока нет полного human-friendly explanation layer
- нет развитой аналитики “почему этот memory brief оказался хорошим или плохим”

## 11. Память не встраивается в агентный runtime, а живет отдельно

### Проблема

Многие memory решения хороши как storage demo, но слабы как реальный runtime-модуль рядом с агентом.

### Почему это больно

- интеграция становится хрупкой
- память не участвует в реальном lifecycle агента
- вся ценность теряется на glue-code

### Как решаем

- строим отдельный `memory-runtime` сервис
- даем adapter contracts
- делаем OpenClaw/BunkerAI integration contour
- добавили `mem0 bridge` как seam между orchestration layer и mem0 ecosystem

### Статус

- `частично решаем`

### Что еще не закрыто

- нужно завершить живые end-to-end интеграции для всех приоритетных клиентов

## 12. Память не измеряется и ее качество нельзя стабилизировать

### Проблема

Без метрик memory system остается “магией”: никто не знает, становится ли она реально лучше или хуже.

### Почему это больно

- невозможно надежно релизить изменения
- ошибки видны слишком поздно
- quality regressions накапливаются незаметно

### Как решаем

- есть `/metrics`
- есть `/v1/observability/stats`
- есть counters по recall/jobs/lifecycle/consolidation
- есть usefulness feedback
- test strategy включает unit/component/integration/contract/golden thinking

### Статус

- `частично решаем`

### Что еще не закрыто

- пока нет полноценного offline eval harness
- нет benchmark suite на recall quality и business scenarios

## 13. Prompt injection и memory poisoning

### Проблема

Агенту можно подбросить вредную информацию, которая потом попадет в память и будет влиять на будущие решения.

### Почему это больно

- компрометируется long-term memory
- agent behavior искажается надолго
- повышается риск системных ошибок

### Как решаем

- на текущем этапе не делаем это главным фокусом архитектуры
- часть риска косвенно снижается через spaces, consolidation и lifecycle

### Статус

- `не решаем пока` как полноценную подсистему

### Комментарий

- это осознанное решение текущего этапа: в `v1` приоритетом зафиксировано качество recall, а не security-first perimeter

## 14. Противоречащие друг другу воспоминания

### Проблема

Система может одновременно помнить:

- старое и новое решение
- старую и новую конфигурацию
- конфликтующие предпочтения

### Почему это больно

- агент теряет консистентность
- растет вероятность неправильных действий
- retrieval начинает подмешивать взаимоисключающие факты

### Как решаем

- сейчас есть только базовый merge контур
- freshness и lifecycle частично уменьшают давление старых записей

### Статус

- `не решаем пока` полноценно

### Что должно появиться позже

- contradiction detection
- supersede/update semantics на уровне фактов
- temporal conflict handling

## 15. Knowledge graph слишком рано усложняет систему

### Проблема

Многие команды пытаются сразу строить graph-memory как обязательное ядро памяти, хотя продукт еще не доказал базовую полезность retrieval.

### Почему это больно

- растет архитектурная сложность
- MVP замедляется
- команда теряет фокус на реальном качестве памяти

### Как решаем

- graph layer признан полезным, но не обязателен в `v1`
- базой выбраны `memory spaces + episodes + memory units + recall orchestration`
- граф остается optional enhancement, а не hard dependency

### Статус

- `решаем` на уровне архитектурного ограничения

## Сводка по статусам

### Уже решаем в рабочем объеме

- забывание между сессиями
- short-term vs long-term separation
- засорение long-term краткосрочным шумом
- forgetting / decay / archive / evict baseline
- shared vs isolated memory model
- usefulness feedback loop

### Частично решаем

- retrieval quality
- deduplication
- stale memory handling
- explainability
- observability as quality system
- integration into real agent runtime

### Пока сознательно не закрываем полностью

- prompt injection / memory poisoning
- contradiction resolution
- полноценный semantic invalidation layer
- learned retrieval optimization

## Практический вывод

Если смотреть на рынок agent-memory систем трезво, то самые дорогие проблемы не в “отсутствии vector DB”, а в четырех вещах:

1. агент не получает нужную память в нужный момент
2. шум попадает в long-term и отравляет recall
3. система не умеет забывать и обновлять знания
4. невозможно понять, почему память сработала хорошо или плохо

Именно эти области сейчас являются ядром нашего проекта.  
Безопасность, contradiction resolution и более глубокая knowledge semantics признаны важными, но на текущем этапе идут после базовой полезности и качества recall.
