# ADR-002: Gate de CI para o openmemory/api via workflow reutilizável no CI Gate

## Status

Aceito

## Data

2026-06-18

## Contexto

Os 39 módulos de teste em `openmemory/api/tests/` não são executados por nenhum workflow de CI. O `ci.yml` existente cobre apenas o SDK Python (`mem0/**`, `tests/**`) e o `ci-gate.yml` é o único status check requerido em branch protection — ele detecta pacotes alterados via `dorny/paths-filter` e invoca workflows reutilizáveis por pacote, agregando o resultado no job `gate`.

Sem gate, qualquer regressão no `openmemory/api` passa despercebida até execução manual local.

## Decisão

Criar um workflow reutilizável `.github/workflows/openmemory-api-ci.yml` que instala as dependências do `openmemory/api`, roda lint e a suíte `pytest` dos 39 módulos, e integrá-lo ao `ci-gate.yml`:

1. Adicionar o filtro `openmemory_api` (paths `openmemory/api/**` + os dois arquivos de workflow) no job `changes`.
2. Adicionar um call job `openmemory-api` que `uses: ./.github/workflows/openmemory-api-ci.yml`.
3. Incluir esse job em `needs` do job `gate`.

Assim o "CI Gate" único continua sendo o status requerido e passa a refletir os testes do `openmemory/api` quando o PR toca esses caminhos.

## Alternativas Consideradas

### Alternativa 1: Adicionar os testes ao `ci.yml` existente

- **Descrição**: Estender o pipeline do SDK Python para também rodar `openmemory/api`.
- **Prós**: Sem novo arquivo de workflow.
- **Contras**: Mistura escopos (SDK mem0 × aplicação openmemory); o `paths-filter` do `ci.yml` é do mem0; polui matriz e dependências.
- **Por que rejeitada**: Acoplamento indevido entre pacotes independentes.

### Alternativa 2: Workflow standalone não integrado ao gate

- **Descrição**: Workflow próprio com trigger `pull_request`, sem entrar no `ci-gate.yml`.
- **Prós**: Mais simples de escrever.
- **Contras**: Não pode ser marcado como required (path-filtered checks ficam pendurados em "Expected"); não bloqueia merge.
- **Por que rejeitada**: O objetivo é justamente barrar merges — exige passar pelo gate agregador.

## Consequências

### Positivas

- Regressões no `openmemory/api` barram o merge automaticamente.
- Padrão idêntico aos demais pacotes do monorepo (consistência operacional).

### Negativas

- Tempo de CI do PR aumenta quando `openmemory/api` é tocado.

### Riscos

- **Testes que exigem serviços (Qdrant/Postgres)**: alguns testes podem precisar de containers. Mitigação: usar `services:` no job ou garantir que dependam de mocks/SQLite (a maioria já usa); marcar os que exigem infra e provê-los como service.

## Notas de Implementação

Matriz Python 3.10–3.12 alinhada ao `ci.yml`. Instalar via `pip install -e` do pacote e extras de teste; rodar `pytest openmemory/api/tests`. Detalhes de comandos pertencem às tasks.
