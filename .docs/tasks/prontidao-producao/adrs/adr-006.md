# ADR-006: Endurecimento para LAN — API key por equipe, secrets gerenciados e rate limit por project

## Status

Aceito

## Data

2026-06-18

## Contexto

O acesso hoje é "trust-on-LAN": `compat_v3` aceita e ignora o token, a identidade só serve para atribuição/audit (`identity.py`), segredos vivem em texto plano no `.env`/compose, e o rate limit do Traefik é global (não por project+hostname, como a arquitetura sugere em §1). Para o alvo LAN, queremos endurecimento **proporcional** — sem mTLS/JWT, que seriam desproporcionais.

## Decisão

1. **Autenticação por equipe**: middleware FastAPI que valida um token por equipe enviado em header, contra um mapa `team → token`. Mantém o contrato MCP e a atribuição por hostname existentes. Período de transição com modo "warn" antes de tornar a validação obrigatória (mitiga risco de adoção do PRD).
2. **Gestão de segredos**: mover tokens e demais segredos para **Docker secrets** (ou arquivo de secrets montado fora do compose versionado); remover valores sensíveis do `.env` versionado. Vault fica adiado (desproporcional para LAN).
3. **Rate limit por project+hostname**: middleware com **Redis sliding-window** (reaproveitando o Redis já presente), aplicando limites de busca e escrita por `(project, hostname)` conforme §1 da arquitetura, substituindo o limite global do Traefik para essas rotas.

## Alternativas Consideradas

### Alternativa 1: mTLS + JWT com expiração

- **Descrição**: Autenticação forte service-to-service e por requisição.
- **Prós**: Padrão para exposição externa.
- **Contras**: Complexidade de PKI/rotação desproporcional para LAN confiável.
- **Por que rejeitada/adiada**: Fora do alvo; reservar para esforço futuro de exposição externa.

### Alternativa 2: Manter rate limit global no Traefik

- **Descrição**: Não introduzir limite por project.
- **Prós**: Zero código novo.
- **Contras**: Não atende à granularidade por project+hostname (gap D8); um consumidor pode monopolizar a cota global.
- **Por que rejeitada**: Não fecha o gap pedido.

### Alternativa 3: HashiCorp Vault

- **Descrição**: Cofre de segredos completo.
- **Prós**: Rotação e auditoria robustas.
- **Contras**: Mais um serviço crítico para operar.
- **Por que rejeitada/adiada**: Docker secrets é suficiente para LAN.

## Consequências

### Positivas

- Fecha D7 (auth/secrets) e D8 (rate limit) de forma proporcional ao risco LAN.
- Reaproveita Redis existente; sem dependências pesadas.

### Negativas

- Clientes MCP passam a precisar configurar a credencial da equipe (mudança visível).

### Riscos

- **Quebra de clientes na virada para obrigatório**: mitigado pelo modo "warn" e comunicação antecipada.
- **Distribuição/rotação de tokens manual**: aceitável para o número de equipes; documentar no runbook.

## Notas de Implementação

Middleware de auth e de rate limit como camadas FastAPI (ou ASGI), ordenadas após o `RequestIdMiddleware`. Mapa de tokens carregado de secret. Limites configuráveis por env, com defaults da §1 (busca 30/min, escrita 60/min, burst 10/10s).
