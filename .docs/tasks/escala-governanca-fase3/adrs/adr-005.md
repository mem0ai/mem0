# ADR-005: Políticas de governança — Config JSON global + tabela de override por projeto

## Status

Aceito

## Data

2026-06-18

## Contexto

A governança precisa de **políticas configuráveis** com **padrão global** e **override por projeto** ([PRD](../_prd.md)): idade/uso para poda (TTL), agenda dos jobs, limiar de similaridade do merge, regra de desempate de contradição, categorias protegidas (pinadas por padrão) e janela de quarentena. O repositório já oferece:

- O modelo **`Config`** (JSON por chave, ex.: `key="main"`, lido/escrito por `routers/config.py`) — já é o canal de configuração global da aplicação.
- A tabela **`ArchivePolicy`** (`criteria_type`/`criteria_id`/`days_to_archive`) — estreita e atualmente **ociosa**.
- A tabela **`projects`** (PK `name`, com `partition_tier`/`shard_key` da Fase 2).

A decisão: onde guardar essas políticas de forma que o padrão global seja fácil de manter e os overrides por projeto sejam **esparsos** (só projetos que precisam divergir).

## Decisão

Modelo de **duas camadas**:

1. **Padrão global em `Config(key="governance")`** — um único documento JSON com todos os knobs da política (ttl por idade e por último acesso, agenda por tipo de job, limiar de similaridade, regra de contradição, categorias protegidas, janela de quarentena). Lido/escrito pelos endpoints de config existentes.

2. **Override por projeto numa nova tabela `governance_policies`** — PK/FK `project_name` (→ `projects.name`), com um JSON **parcial** que sobrescreve apenas os campos divergentes do global. Projetos sem linha herdam 100% do global.

A **resolução efetiva** é feita por um helper que faz *merge* (global ⟵ override) e entrega a política aplicável a um projeto. A tabela `ArchivePolicy` ociosa **não é reutilizada** (schema estreito demais); permanece como está ou é removida em limpeza futura (fora de escopo).

## Alternativas Consideradas

### Alternativa 1: Estender `ArchivePolicy`

- **Descrição**: Generalizar a tabela ociosa para cobrir global e por projeto via `criteria_type`/`criteria_id`.
- **Prós**: Reaproveita tabela existente.
- **Contras**: Schema atual cobre só `days_to_archive`; precisaria crescer muito (vários campos ou um JSON), descaracterizando o modelo original; `criteria_type/criteria_id` genéricos são menos legíveis que `project_name` explícito.
- **Por que rejeitada**: O ganho de "reusar" é ilusório — exigiria reescrever o schema; mais limpo modelar a política nova explicitamente.

### Alternativa 2: Colunas de política na tabela `projects`

- **Descrição**: Adicionar `ttl_days`, `consolidation_schedule`, etc. direto em `projects`, com um registro global à parte.
- **Prós**: Simples para poucos campos.
- **Contras**: Mistura **catálogo** (identidade/tamanho do projeto) com **política**; engessa o schema a cada novo knob; todo projeto carrega colunas mesmo sem divergir do global.
- **Por que rejeitada**: Acoplamento indevido e baixa flexibilidade conforme as políticas evoluem.

### Alternativa 3: Tudo em `Config` (global e por projeto no mesmo JSON)

- **Descrição**: Um único JSON com um mapa `por_projeto`.
- **Prós**: Sem tabela nova.
- **Contras**: Documento cresce sem limite com o nº de projetos; concorrência de escrita no mesmo blob; difícil consultar/auditar override de um projeto isolado.
- **Por que rejeitada**: Não escala para centenas de projetos nem permite consulta/edição granular.

## Consequências

### Positivas

- Padrão global trivial de manter (um JSON), no canal de config já existente.
- Overrides esparsos e explícitos por projeto, consultáveis e auditáveis individualmente.
- Schema flexível (JSON) acomoda novos knobs sem migração a cada mudança de política.

### Negativas

- Lógica de *merge* global⟵override a implementar e testar (precedência, campos ausentes).
- Validação dos JSONs de política fica na aplicação (sem constraints fortes de coluna).

### Riscos

- **Política inválida/ inconsistente quebrando jobs.** _Mitigação:_ schema de validação (Pydantic) na escrita dos endpoints + valores-padrão conservadores no merge quando um campo falta.
- **Divergência entre o que o operador vê e o que o job aplica.** _Mitigação:_ um único helper de resolução efetiva, usado tanto pela API (preview) quanto pelo worker.

## Notas de Implementação

- `Config(key="governance")`: documento global; reutilizar `get_config_from_db`/setter de `routers/config.py`.
- Nova tabela `governance_policies(project_name PK/FK, overrides JSON, updated_at)` + Alembic.
- Helper `resolve_policy(project) -> EffectivePolicy` (merge global+override) consumido pelo `governance-worker` e pelos endpoints `/admin/governance/policies`.
- Validação por modelos Pydantic dos campos da política.
