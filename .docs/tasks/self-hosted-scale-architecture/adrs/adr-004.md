# ADR-004: Borda via reverse proxy (Traefik/Nginx) com rate limit, circuit breaker e sticky sessions para MCP SSE

> Não confundir com o ADR-004 da task `memoria-central-compartilhada` (fila assíncrona). Este ADR pertence à task de escala.

## Status

Aceito

## Data

2026-06-16

## Contexto

Com 200 devs e dezenas de agentes, rajadas de busca (context-loader dispara 2–4 queries/sessão) e agentes em loop podem saturar o cluster. O PRD exige proteção de borda (rate limit + circuit breaker) e capacidade de leitura escalável com múltiplas réplicas da API atrás de um ponto de entrada único. O ambiente é Docker Swarm / VMs (sem Istio/K8s, portanto sem service mesh nativo).

A API MCP usa **SSE e Streamable HTTP** ([mcp_server.py](openmemory/api/app/mcp_server.py)). SSE mantém conexões longas e estado de sessão, exigindo **afinidade de sessão (sticky sessions)** quando há múltiplas réplicas — caso contrário, mensagens da mesma sessão podem cair em réplicas diferentes.

## Decisão

- Usar um **reverse proxy (Traefik ou Nginx)** à frente das réplicas da API como camada de borda.
- Configurar **rate limiting** por `project` + `hostname` (busca) e por `project` (escrita), com limites de burst por cliente.
- Configurar **circuit breaker** abrindo sob taxa de erro alta para Qdrant/embedding.
- Habilitar **sticky sessions** (afinidade por cookie/hash) para as conexões MCP SSE, garantindo que a sessão permaneça na mesma réplica.
- Balancear as réplicas da API (stateless quanto a dados; estado de sessão SSE tratado por afinidade).

## Alternativas Consideradas

### Alternativa 1: Middleware FastAPI + Redis (sliding window)

- **Descrição**: Rate limit dentro da própria API com contadores em Redis.
- **Prós**: Zero infra nova; controle fino por project/hostname; reusa o Redis do cache.
- **Contras**: Roda dentro da app (não protege antes de a requisição chegar); não resolve sticky sessions nem circuit breaker de borda.
- **Por que rejeitada como camada principal**: Proteção precisa começar antes da app; o proxy também resolve LB + afinidade SSE. (Pode complementar com limites finos por project.)

### Alternativa 2: API Gateway dedicado (Kong/APISIX)

- **Descrição**: Gateway com rate limit por token, circuit breaker e observabilidade integrados.
- **Prós**: Recursos avançados; rate limit por token.
- **Contras**: Peso operacional alto para o time de infra em Swarm/VMs.
- **Por que rejeitada**: Excesso de complexidade para o MVP; reverse proxy atende as metas (YAGNI).

## Consequências

### Positivas

- Proteção de borda antes da app; rajadas e loops não derrubam o serviço.
- LB + afinidade SSE resolvidos na mesma camada, simples de operar em Swarm.

### Negativas

- Sticky sessions reduzem a uniformidade de distribuição de carga entre réplicas.
- Configuração de rate limit por `project`/`hostname` no proxy pode exigir extração desses campos da rota/headers.

### Riscos

- **MCP através de proxy ainda é terreno pouco padronizado**: testar propagação de auth e afinidade SSE no piloto.
- **Limites mal calibrados** podem bloquear uso legítimo: começar permissivo e ajustar por métricas.

## Notas de Implementação

- Limites sugeridos (calibrar em carga): busca 30 req/min por `project`+`hostname`; escrita 60 req/min por `project`; burst 10 req/10s por cliente.
- Afinidade SSE por cookie de sessão ou hash de origem.
- Opcional: complementar com middleware Redis para limites finos por `project` não expressáveis no proxy.
