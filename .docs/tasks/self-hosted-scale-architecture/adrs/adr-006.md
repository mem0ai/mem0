# ADR-006: Instalação automatizada via stack único + bootstrap script

## Status

Aceito

## Data

2026-06-16

## Contexto

Um requisito do projeto é a **instalação/configuração simplificada e automatizada** de todos os recursos necessários. O deploy atual é um `docker-compose.yml` quase autocontido com auto-detecção de modelo no install. A arquitetura em escala, porém, introduz muitas peças novas (PostgreSQL + PgBouncer, Redis, dois serviços Ollama dedicados, workers em processo separado, reverse proxy, Prometheus/Grafana). Sem automação, a instalação passaria de "um comando" para um roteiro manual extenso e propenso a erro — o oposto do requisito.

O ambiente-alvo é Docker Swarm / VMs ([ADR-001](adr-001.md)), sem Kubernetes. O repo já possui base reutilizável: fluxo de provisionamento, endpoint `/discovery` e detecção de modelo no install (ADR-006 da task `memoria-central-compartilhada`).

## Decisão

Entregar a instalação como **stack único + bootstrap script**:

- **Arquivo de stack Swarm** (e instruções/compose equivalente para VMs) que sobe todos os serviços com **um comando** (`docker stack deploy`), com healthchecks e ordem de dependência entre serviços.
- **Bootstrap script idempotente** que, na primeira subida:
  - provisiona o banco PostgreSQL e **roda as migrations Alembic automaticamente**;
  - cria/garante os **índices de payload no Qdrant** necessários;
  - aplica a **configuração via template de variáveis de ambiente** (endpoints dos serviços dedicados, credenciais, TTLs de cache, limites de rate), decidindo o **modo de provisionamento** (escala vs dev/single-host) e o **backend de inferência detectado** (Ollama via `/api/tags` ou llama.cpp via `llama-server` OpenAI-compatible, [ADR-002](adr-002.md));
  - valida a saúde dos componentes (`/health`) antes de liberar a borda.
- **Configuração por env templating** centralizada (um `.env`/secret de stack), evitando edição manual espalhada.

A **migração de dados SQLite → PostgreSQL** é tratada como passo de bootstrap opcional/documentado, não totalmente automática no MVP (ver Consequências).

## Alternativas Consideradas

### Alternativa 1: Setup wizard guiado (CLI/UI interativo)

- **Descrição**: Assistente que detecta recursos, pergunta o mínimo e gera config + sobe serviços.
- **Prós**: Experiência mais amigável para o operador.
- **Contras**: Mais código a manter; superfície de UI/CLI interativa; menos reproduzível em automação/CI.
- **Por que rejeitada**: Para o MVP, stack declarativo + script é mais simples, versionável e reproduzível (YAGNI); um wizard pode vir depois.

### Alternativa 2: Instalação manual documentada (runbook)

- **Descrição**: Apenas documentar os passos para o operador executar manualmente.
- **Prós**: Zero código de automação.
- **Contras**: Viola o requisito de instalação simplificada/automatizada; propenso a erro com tantos componentes.
- **Por que rejeitada**: Não atende ao requisito.

## Consequências

### Positivas

- Instalação reproduzível com um comando, apesar do número de componentes.
- Migrations e índices aplicados automaticamente reduzem erro humano.
- Config centralizada por template facilita ambientes (dev/prod) e o modo duplo do ADR-002.

### Negativas

- Manter o stack + script atualizados conforme os componentes evoluem.
- Migração de dados SQLite→PostgreSQL não 100% automática no MVP (passo guiado).

### Riscos

- **Ordem de subida/healthcheck mal configurada**: mitigar com `depends_on`/healthchecks e o gate de `/health` antes da borda.
- **Templating de config inconsistente entre ambientes**: mitigar com um único arquivo de variáveis versionado e validação no bootstrap.

## Notas de Implementação

- Migrations Alembic já são driver-agnostic ([ADR-003](adr-003.md)); o bootstrap apenas dispara `alembic upgrade head` contra o `DATABASE_URL`.
- Reusar/expandir o fluxo de provisionamento e `/discovery` existentes para registrar endpoints dos serviços dedicados.
- O bootstrap define o modo (escala vs dev) gravando ou não os `base_url` explícitos do Ollama ([ADR-002](adr-002.md)).
