# ADR-003: Fila durável em PostgreSQL com FOR UPDATE SKIP LOCKED e workers em processo separado

> Não confundir com o ADR-003 da task `memoria-central-compartilhada` (escopo por `project`). Este ADR pertence à task de escala.

## Status

Aceito

## Data

2026-06-16

## Contexto

O ADR-004 (task `memoria-central-compartilhada`) estabeleceu a fila interna de escrita com ack imediato e worker de background, persistida em SQLite. Hoje o `WriteWorker` roda **embutido no mesmo processo FastAPI** (`@app.on_event("startup")` em [main.py](openmemory/api/main.py)) com `max_concurrency=2` fixo, e o `dequeue` ([write_queue.py](openmemory/api/app/utils/write_queue.py)) faz `SELECT ... LIMIT` seguido de marcação para `processing` **sem lock de linha**.

Para escalar a escrita para 200 devs é preciso: (a) múltiplos workers concorrentes; (b) backend que suporte centenas de conexões; (c) dequeue seguro entre workers concorrentes. Com SQLite e o dequeue atual, dois workers pegariam os mesmos jobs (corrida) e o SQLite trava sob muitas conexões.

A camada de dados já é agnóstica: `DATABASE_URL` é o único switch ([database.py](openmemory/api/app/database.py)) e o Alembic é driver-agnostic.

## Decisão

- **Migrar a fila para PostgreSQL** via `DATABASE_URL`, condicionando o `check_same_thread` (específico de SQLite) ao dialeto.
- **Tornar o `dequeue` seguro para concorrência** usando `SELECT ... FOR UPDATE SKIP LOCKED` (com `with_for_update(skip_locked=True)`), garantindo que cada job seja entregue a um único worker.
- **Extrair o `WriteWorker` para um processo/serviço separado** (novo entrypoint chamando `write_worker.run()`), removendo o startup hook embutido (gated por env para compatibilidade local).
- **Tornar `max_concurrency` e número de réplicas configuráveis** (env/constructor), escalonados manualmente em Swarm/VMs.
- Manter o ack imediato (`queued` + `job_id`) e a lógica de retry/backoff existentes (ADR-004 original).

## Alternativas Consideradas

### Alternativa 1: Redis Streams + consumer groups

- **Descrição**: Substituir a fila SQL por Redis Streams com ack/claim automáticos.
- **Prós**: Menor latência, throughput maior, consumer groups nativos.
- **Contras**: Durabilidade frágil se mal configurado; reescreve a camada de fila e a auditoria; novo modelo mental.
- **Por que rejeitada**: O PostgreSQL reusa `WriteQueueJob`/auditoria existentes com mudança mínima e ACID; throughput-alvo (20–100 writes/s) é atingível.

### Alternativa 2: Manter SQLite com worker único embutido

- **Descrição**: Não migrar; apenas aumentar `max_concurrency`.
- **Prós**: Zero mudança de infra.
- **Contras**: SQLite trava sob centenas de conexões MCP; worker preso ao ciclo de vida da API; sem escala horizontal de escrita.
- **Por que rejeitada**: Apenas deslocaria o gargalo de leitura para a escrita/conexões.

## Consequências

### Positivas

- Escrita escala por número de workers e concorrência por pod, independente da API.
- Dequeue seguro entre workers; nenhum job duplicado ou perdido.
- PostgreSQL suporta as conexões da API + workers (com PgBouncer).

### Negativas

- Novo processo/serviço a operar e observar.
- Migração de dados SQLite → PostgreSQL exige janela planejada.

### Riscos

- **Migração sob sistema em uso**: mitigar com janela enquanto a base é pequena.
- **Saturação de conexões PostgreSQL**: mitigar com PgBouncer (transaction mode) entre API/workers e o banco.

## Notas de Implementação

- Conectar API e workers via PgBouncer (transaction mode), não direto no PostgreSQL.
- `dequeue` com `with_for_update(skip_locked=True)` dentro de transação; manter o índice `idx_write_queue_status_created`.
- Entrypoint do worker: script dedicado (`python -m app.workers.write_worker`) iniciando `WriteWorker.run()`.
- Manter o caminho síncrono do `compat_v3` para hooks do plugin, conforme já existente.
