# ADR-002: Embedding e LLM como serviços Ollama dedicados e replicados, plugados por base_url

## Status

Aceito

## Data

2026-06-16

## Contexto

No deploy atual, o Ollama roda inline no host e atende embedding e extração LLM no mesmo processo/recurso. Sob a carga de 200 devs, o embedding síncrono antes de cada busca é o gargalo #1 de latência. A camada de embedder/LLM do mem0 já é configurável: `EmbedderFactory`/`LlmFactory` ([factory.py](mem0/utils/factory.py)) resolvem o provider por configuração, e os providers `ollama` ([ollama.py](mem0/embeddings/ollama.py)) e `openai`-compatible aceitam `ollama_base_url`/`openai_base_url`. O cliente é instanciado em [memory.py](openmemory/api/app/utils/memory.py) a partir de env vars + config no banco.

A equipe optou por **manter Ollama** (em vez de migrar para TEI/vLLM) para minimizar a mudança de stack e preservar o modo self-hosted local-only.

## Decisão

- **Separar embedding e LLM em dois serviços de inferência local dedicados** (Ollama ou llama.cpp), não inline no host, cada um com suas réplicas, acessíveis por URL própria.
- **Não compartilhar o mesmo recurso/GPU** entre embedding e extração LLM, evitando contenção.
- **Reuso integral da camada existente**: apenas configurar `EMBEDDER_PROVIDER=ollama` + `OLLAMA_BASE_URL` (embedding) e o provider/`base_url` do LLM apontando para os serviços dedicados. **Nenhum código novo de embedder/LLM** é necessário.
- **Modo duplo decidido no provisionamento (instalação)**: a **auto-detecção de backend/modelo é um passo de instalação**, não de runtime. No install, a detecção no host permanece para o cenário **dev/single-host**, gravando a configuração detectada. Para o deploy **em escala/produção**, o provisionamento configura os endpoints **explicitamente** para os serviços dedicados, e essa configuração tem precedência — a detecção no host não é executada. Em runtime, `_fix_ollama_urls()` segue resolvendo `localhost`→`host.docker.internal` apenas quando o endpoint não for explícito (modo dev). Nenhuma trilha exige reescrita do mem0 — é decisão de provisionamento/env.
- **Auto-detecção multi-backend (Ollama + llama.cpp)**: a detecção no install sonda **dois backends locais**: Ollama (via `/api/tags`) e **llama.cpp** (via o `llama-server`, que expõe API **OpenAI-compatible** em `/v1/models`, `/v1/embeddings`, `/v1/chat/completions`). Ollama pluga pelo provider `ollama`; llama.cpp pluga pelo provider **`openai` com `openai_base_url`** apontando para o `llama-server` — ambos **sem código novo de provider** no mem0. O install registra qual backend foi detectado (ou configurado explicitamente em produção) e grava o provider/`base_url` correspondente.
- Como Ollama não faz batching contínuo, o **cache Redis ([ADR-005](adr-005.md)) é a alavanca principal** de latência de leitura, compensando o throughput inferior.

## Alternativas Consideradas

### Alternativa 1: TEI / vLLM / Infinity como serviço de embedding

- **Descrição**: Trocar Ollama por um servidor de embeddings com batching contínuo (TEI para baixa latência, vLLM para throughput).
- **Prós**: Maior throughput/latência sob concorrência de frota; batching nativo.
- **Contras**: Novo componente no stack; novo modelo/formato a operar; desvio do Ollama já dominado pela equipe.
- **Por que rejeitada**: A equipe priorizou menor mudança de stack; o cache Redis cobre boa parte do ganho de latência sem trocar o motor.

### Alternativa 2: Manter Ollama inline (status quo)

- **Descrição**: Não extrair o Ollama para serviço dedicado.
- **Prós**: Zero mudança.
- **Contras**: Mantém o gargalo #1; embedding compete com extração; não escala horizontalmente.
- **Por que rejeitada**: Viola o objetivo central de remover o gargalo de leitura.

## Consequências

### Positivas

- Embedding deixa de competir com extração; ambos escalam por réplicas.
- Mudança mínima de código (apenas configuração).
- Preserva self-hosted local-only.

### Negativas

- Ollama sem batching contínuo tem throughput inferior a TEI/vLLM; dependência forte do cache.
- Dois serviços Ollama a operar e dimensionar (hardware/GPU separados).

### Riscos

- **Throughput de embedding insuficiente em pico**: mitigar com réplicas + cache agressivo; monitorar `embed_cache_hit_rate` e latência.
- **Dimensionamento de hardware**: necessidade de GPU/recursos para dois serviços; registrado como dependência técnica.

## Notas de Implementação

- Configurar via env (`OLLAMA_BASE_URL`, `EMBEDDER_PROVIDER`, `LLM_PROVIDER`, `LLM_BASE_URL`) apontando para os serviços dedicados.
- Remover o Ollama do host do caminho de leitura; o cliente mem0 passa a chamar o serviço dedicado por HTTP.
- Usar `embed_batch` onde aplicável no worker de escrita.
