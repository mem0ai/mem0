# ADR-005: Cache Redis de embedding e de resultado de busca com invalidação por escrita

## Status

Aceito

## Data

2026-06-16

## Contexto

Com o embedding em Ollama escalado ([ADR-002](adr-002.md)) — sem batching contínuo — o cache é a principal alavanca de latência de leitura. O caminho de busca atual ([mcp_server.py](openmemory/api/app/mcp_server.py)) calcula o embedding inline a cada chamada e em seguida consulta o Qdrant, sem reaproveitamento. Agentes repetem muitas queries semelhantes por sessão, o que torna o cache de alto retorno (hit rate esperado de 40–70% em uso real de IDE).

## Decisão

Introduzir um **cache Redis em duas camadas** no caminho de leitura:

- **Cache de embedding**: chave `embed:v1:{model}:{sha256(query_normalizada)}`, TTL 1–24h. A query é normalizada (trim, lowercase, collapse whitespace) antes do hash. Hit → pula o Ollama.
- **Cache de resultado de busca**: chave `search:v1:{project}:{sha256(query)}:{top_k}:{filter_hash}`, TTL curto (~5 min). Hit → pula embedding e Qdrant.
- **Invalidação**: ao processar uma escrita de um `project`, invalidar o cache de resultado de busca daquele `project` (o cache de embedding não precisa de invalidação, pois o vetor da query não muda).

O mesmo cluster Redis hospeda ambas as camadas; contadores de rate limit, se necessários, ficam fora do escopo deste ADR (ver [ADR-004](adr-004.md)).

## Alternativas Consideradas

### Alternativa 1: Apenas cache de embedding

- **Descrição**: Cachear só o vetor, sem cache de resultado.
- **Prós**: Mais simples; sem invalidação.
- **Contras**: Ainda paga a busca no Qdrant a cada chamada; ganho parcial.
- **Por que rejeitada**: Com Ollama escalado, maximizar o ganho de leitura justifica também o cache de resultado.

### Alternativa 2: Sem cache (escalar só Ollama/Qdrant)

- **Descrição**: Contar apenas com réplicas.
- **Prós**: Sem camada extra.
- **Contras**: Ollama sem batching satura rápido; latência não atinge as metas sub-segundo.
- **Por que rejeitada**: Não atinge a métrica de p99 do PRD.

## Consequências

### Positivas

- Latência sub-segundo no caminho quente; absorve a repetição de queries dos agentes.
- Compensa o throughput inferior do Ollama sem trocar o motor.

### Negativas

- Risco de servir resultado levemente desatualizado dentro do TTL de 5 min (mitigado pela invalidação por escrita).
- Mais um componente (Redis) a operar e monitorar.

### Riscos

- **Invalidação incompleta**: se a invalidação por `project` falhar, resultados ficam stale até o TTL; mitigar com TTL curto e métrica de hit rate.
- **Consumo de memória do Redis**: dimensionar TTLs e monitorar uso.

## Notas de Implementação

- Normalização da query centralizada e reutilizada pelas duas camadas.
- Invalidação acionada no `WriteWorker` após upsert no Qdrant (passo já existente no fluxo de escrita).
- Métricas: `embed_cache_hit_rate`, `search_cache_hit_rate` (ver observabilidade).
