# ADR-001: Entrega integrada das Fases 0+1 com caminho de leitura priorizado

## Status

Aceito

## Data

2026-06-16

## Contexto

O deploy padrão do OpenMemory/Mem0 (`docker-compose.yml`: Qdrant single-node, Ollama inline, SQLite, write worker embutido com `max_concurrency=2`) não sustenta a carga prevista de 200 devs e dezenas de agentes MCP. O sintoma mais crítico relatado é a **latência de busca na sessão**: o context-loader dispara 2–4 buscas por sessão e cada uma faz embedding inline (Ollama compartilhado) antes de consultar o Qdrant, gerando segundos de espera no início da interação com o agente.

A proposta técnica de referência (`self-hosted-scale-architecture.md`) define quatro fases de migração:

- **Fase 0 — Fundação**: PostgreSQL + PgBouncer, extração dos write workers para serviço separado, métricas básicas.
- **Fase 1 — Desacoplamento**: serviço de embedding dedicado + cache Redis, rate limiting na borda, API replicada atrás de load balancer, remoção do Ollama do caminho de leitura.
- **Fase 2 — Particionamento Qdrant** e **Fase 3 — Escala e governança**: fora do escopo deste MVP.

A decisão é sobre **como sequenciar a entrega das Fases 0+1**, dado que: (a) o objetivo de negócio é desbloquear a adoção pelos 200 devs; (b) a dor #1 é latência de leitura; (c) a métrica de sucesso combina latência p99 e adoção ativa sem incidentes; (d) o ambiente de deploy é Docker Swarm / VMs (sem Kubernetes, portanto sem HPA).

## Decisão

Tratar as **Fases 0+1 como um MVP único e coeso**, ordenando o rollout interno para que o **caminho de leitura** (serviço de embedding dedicado, cache Redis de embedding, API replicada atrás de load balancer e rate limit na borda) seja priorizado e validado primeiro, assentado sobre a fundação durável (PostgreSQL + PgBouncer + write workers desacoplados) entregue no mesmo ciclo.

O caminho de escrita migra para PostgreSQL como fila durável e pool de workers configurável no mesmo MVP, garantindo que a remoção do gargalo de leitura não exponha um novo gargalo de escrita/conexões sob a carga dos 200 devs.

## Alternativas Consideradas

### Alternativa 1: Fundação-primeiro estrita (dois releases sequenciais)

- **Descrição**: Release 1 entrega apenas a Fase 0 (PostgreSQL, PgBouncer, workers separados, métricas); Release 2 entrega a Fase 1 (embedding dedicado, cache, latência).
- **Prós**: Menor risco por release; cada etapa é validável isoladamente; fundação sólida antes de otimizar.
- **Contras**: O ganho de latência de leitura — a dor #1 dos devs — só chega no segundo release. A adoção pode estagnar e a percepção de valor atrasa.
- **Por que rejeitada**: Não ataca a dor principal cedo o suficiente, comprometendo a métrica de adoção do MVP.

### Alternativa 2: Leitura cirúrgica mantendo SQLite

- **Descrição**: Adiar a migração para PostgreSQL e entregar apenas os componentes do caminho de leitura (embedding dedicado, cache, API replicada, rate limit) sobre o stack atual com SQLite.
- **Prós**: Caminho mais rápido até o alívio percebido de latência; menor esforço inicial.
- **Contras**: Mantém o gargalo de SQLite com centenas de conexões MCP e a escrita acoplada; sob 200 devs simultâneos há risco real de instabilidade e perda de frescor das escritas.
- **Por que rejeitada**: Conflita com a métrica de adoção sem incidentes; a otimização de leitura apenas deslocaria o gargalo para a escrita/conexões.

## Consequências

### Positivas

- Devs percebem o ganho de latência de leitura cedo, sustentando a adoção.
- A fundação durável (PostgreSQL + workers desacoplados) entra junto, evitando deslocar o gargalo para a escrita.
- Um único ciclo de MVP entrega valor de negócio mensurável em ambas as dimensões (p99 + adoção).

### Negativas

- Exige coordenar fundação e caminho de leitura no mesmo ciclo, com esforço inicial e superfície de teste maiores que um release fatiado.
- Maior número de componentes novos validados simultaneamente.

### Riscos

- **Risco**: Complexidade de orquestrar múltiplos serviços novos em Swarm/VMs ao mesmo tempo. **Mitigação**: rollout interno faseado dentro do MVP (fundação validada antes de redirecionar o tráfego de leitura) e métricas básicas desde o início.
- **Risco**: Migração de dados SQLite → PostgreSQL sob sistema em uso. **Mitigação**: janela de migração planejada com a base de dados existente ainda pequena.

## Notas de Implementação

- Sem Kubernetes/HPA: réplicas e workers escalam manualmente via Swarm/VMs no MVP; autoscaling fica para fase futura.
- Reaproveitar `WriteQueueJob` e migrations Alembic existentes como base para a fila PostgreSQL.
- Manter o contrato MCP (`search_memory`, `add_memories`) inalterado; trocar apenas os backends.
- Particionamento do Qdrant (coleções por project) e governança (dream/TTL) ficam explicitamente fora deste MVP.
