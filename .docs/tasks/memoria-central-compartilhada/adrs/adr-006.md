# ADR-006: Detecção de LLMs/embedders locais na instalação via Ollama

> **Idioma:** escreva todo o conteúdo deste arquivo em **português brasileiro (PT-BR)**.

## Status

Aceito

## Data

2026-06-15

## Contexto

A decisão anterior (registrada no TechSpec) era **não fixar um modelo default** e exigir que o admin informasse o LLM/embedder locais na instalação. Isso transfere ao admin o ônus de saber exatamente quais modelos estão instalados e digitar seus nomes corretamente — propenso a erro e atrito.

O provider **Ollama** já é suportado pelo projeto e expõe o endpoint `GET /api/tags`, que lista os modelos instalados localmente. Logo, o instalador pode **descobrir automaticamente** os modelos disponíveis na máquina/rede e oferecê-los para seleção, sem mudar o stack.

## Decisão

O instalador/setup deve **consultar o Ollama (`/api/tags`) para detectar os modelos locais disponíveis** e **listá-los para o admin escolher** o LLM e o embedder a usar. A seleção alimenta a configuração de runtime do mem0 (`llm`/`embedder` providers = `ollama`, com o `model` escolhido). Não há auto-seleção nem download automático no MVP: o admin decide entre os modelos já instalados.

## Alternativas Consideradas

### Alternativa 1: Detectar e auto-selecionar

- **Descrição**: O setup detecta e escolhe automaticamente um LLM e um embedder (por heurística), com override.
- **Prós**: Mais automático; "funciona de primeira".
- **Contras**: Heurística pode escolher modelo inadequado (tamanho/latência/qualidade); menos controle do admin.
- **Por que rejeitada**: O admin conhece melhor o hardware e o trade-off latência/qualidade; preferir controle explícito.

### Alternativa 2: Detectar + opção de baixar (pull)

- **Descrição**: Listar instalados e, se nenhum adequado, oferecer baixar um modelo recomendado via Ollama durante a instalação.
- **Prós**: Mais completo; cobre máquina sem modelos.
- **Contras**: Mais a construir; depende de download (tempo/banda/conectividade ao registro de modelos).
- **Por que rejeitada (para o MVP)**: Escopo extra; pode ser item de fase futura. O MVP assume que há modelos locais instalados.

## Consequências

### Positivas

- Menos atrito e erro na instalação: o admin escolhe de uma lista real, não digita nomes.
- Reusa o Ollama já suportado; nenhuma nova dependência.
- Mantém o controle do admin sobre o trade-off latência/qualidade.

### Negativas

- Depende do Ollama estar acessível no momento do setup.
- Não cobre o caso de máquina sem modelos instalados (fica como pré-requisito/fase futura).

### Riscos

- **Ollama indisponível no setup**: a detecção falha; mitigar com fallback para entrada manual do nome do modelo e mensagem clara.
- **Embedder ausente na lista**: nem todo modelo serve como embedder; mitigar sinalizando modelos de embedding compatíveis e validando a escolha.

## Notas de Implementação

- Adicionar um passo de setup que faz `GET {OLLAMA_BASE_URL}/api/tags`, parseia a lista de modelos e a apresenta para seleção (LLM e embedder).
- Persistir a escolha na configuração de runtime usada por `Memory.from_config` (providers `ollama`).
- Fallback para entrada manual caso a detecção falhe.
